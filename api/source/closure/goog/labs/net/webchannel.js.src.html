15/**
16 * @fileoverview The API spec for the WebChannel messaging library.
17 *
18 * Similar to HTML5 WebSocket and Closure BrowserChannel, WebChannel
19 * offers an abstraction for point-to-point socket-like communication between
20 * a browser client and a remote origin.
21 *
22 * WebChannels are created via <code>WebChannel</code>. Multiple WebChannels
23 * may be multiplexed over the same WebChannelTransport, which represents
24 * the underlying physical connectivity over standard wire protocols
25 * such as HTTP and SPDY.
26 *
27 * A WebChannels in turn represents a logical communication channel between
28 * the client and server end point. A WebChannel remains open for
29 * as long as the client or server end-point allows.
30 *
31 * Messages may be delivered in-order or out-of-order, reliably or unreliably
32 * over the same WebChannel. Message delivery guarantees of a WebChannel is
33 * to be specified by the application code; and the choice of the
34 * underlying wire protocols is completely transparent to the API users.
35 *
36 * Client-to-client messaging via WebRTC based transport may also be support
37 * via the same WebChannel API in future.
38 *
39 * Note that we have no immediate plan to move this API out of labs. While
40 * the implementation is production ready, the API is subject to change
41 * (addition):
42 * 1. Completely new W3C APIs for Web messaging may emerge in near future.
43 * 2. New programming models for cloud (on the server-side) may require
44 * new APIs to be defined.
45 * 3. WebRTC DataChannel alignment
46 * Lastly, we also want to white-list all internal use cases. As a general rule,
47 * we expect most applications to rely on stateless/RPC services.
48 *
49 */

58/**
59 * A WebChannel represents a logical bi-directional channel over which the
60 * client communicates with a remote server that holds the other endpoint
61 * of the channel. A WebChannel is always created in the context of a shared
62 * {@link WebChannelTransport} instance. It is up to the underlying client-side
63 * and server-side implementations to decide how or when multiplexing is
64 * to be enabled.
65 *
66 * @interface
67 * @extends {EventTarget}

72/**
73 * Configuration spec for newly created WebChannel instances.
74 *
75 * WebChannels are configured in the context of the containing
76 * {@link WebChannelTransport}. The configuration parameters are specified
77 * when a new instance of WebChannel is created via {@link WebChannelTransport}.
78 *
79 * messageHeaders: custom headers to be added to every message sent to the
80 * server. This object is mutable, and custom headers may be changed, removed,
81 * or added during the runtime after a channel has been opened.
82 *
83 * messageUrlParams: custom url query parameters to be added to every message
84 * sent to the server. This object is mutable, and custom parameters may be
85 * changed, removed or added during the runtime after a channel has been opened.
86 *
87 * clientProtocolHeaderRequired: whether a special header should be added to
88 * each message so that the server can dispatch webchannel messages without
89 * knowing the URL path prefix. Defaults to false.
90 *
91 * concurrentRequestLimit: the maximum number of in-flight HTTP requests allowed
92 * when SPDY is enabled. Currently we only detect SPDY in Chrome.
93 * This parameter defaults to 10. When SPDY is not enabled, this parameter
94 * will have no effect.
95 *
96 * supportsCrossDomainXhr: setting this to true to allow the use of sub-domains
97 * (as configured by the server) to send XHRs with the CORS withCredentials
98 * bit set to true.
99 *
100 * testUrl: the test URL for detecting connectivity during the initial
101 * handshake. This parameter defaults to "/<channel_url>/test".
102 *
103 *
104 * @typedef {{
105 * messageHeaders: (!Object<string, string>|undefined),
106 * messageUrlParams: (!Object<string, string>|undefined),
107 * clientProtocolHeaderRequired: (boolean|undefined),
108 * concurrentRequestLimit: (number|undefined),
109 * supportsCrossDomainXhr: (boolean|undefined),
110 * testUrl: (string|undefined)
111 * }}
112 */
113goog.net.WebChannel.Options;
